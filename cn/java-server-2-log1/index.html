1、使用token

先定义了JwtUtils工具类，具体实现如下。

（1）生成token

JWT 最常见的几种签名算法(JWA)有HS256(HMAC-SHA256) 、RS256(RSA-SHA256) 还有 ES256(ECDSA-SHA256)。这里我们使用的是HS256。

  1. HS256 使用同一个「secret_key」进行签名与验证（对称加密）。一旦 secret_key 泄漏，就毫无安全性可言了。
    • 因此 HS256 只适合集中式认证，签名和验证都必须由可信方进行。
    • 传统的单体应用广泛使用这种算法，但是请不要在任何分布式的架构中使用它！
  2. RS256 是使用 RSA 私钥进行签名，使用 RSA 公钥进行验证。公钥即使泄漏也毫无影响，只要确保私钥安全就行。
    • RS256 可以将验证委托给其他应用，只要将公钥给他们就行。
  3. ES256 和 RS256 一样，都使用私钥签名，公钥验证。算法速度上差距也不大，但是它的签名长度相对短很多（省流量），并且算法强度和 RS256 差不多。

从代码中可以看出，token中所携带的信息有用户id、用户名、现在的时间、token过期时间还有密钥。

3、跨域

日后，我们的项目是一定要跑在服务器上的，而前后端交流也必然少不了跨域。

据我所知，只要协议、端口、域名三者任意一个不同，就会产生跨域问题。而同样，在前端、服务器、后端三种任意一个添加跨域设置即可实现跨域。

前端：

对于前端有9种方法：

1、 通过jsonp跨域
2、 document.domain + iframe跨域
3、 location.hash + iframe
4、 window.name + iframe跨域
5、 postMessage跨域
6、 跨域资源共享（CORS）
7、 nginx代理跨域
8、 nodejs中间件代理跨域
9、 WebSocket协议跨域

服务器：

可以设置反向代理，将本机设为反向代理服务器，将url的请求转到另一个url处。我们的服务器使用的是nginx，就以nginx为例。

6、增删改查时踩过的坑

（1）空指针

在update前，首先要判断是否已有该信息，若没有，则空指针。

在delete后，再查询信息时，要先判断是否为空，否则也会有空指针。

（2）其他

<1>update覆盖bug，要逐一确认是否有新参数，将旧值覆盖。

<2>BCryptPasswordEncoder这个类存在于springboot-starter-security和spring-security-core这两个依赖中。但是，导入前者会导致任何请求都无权限，经我debug测试，它甚至会在我定义的filter前把请求给拦截下来；后一个依赖，是正常符合逻辑的。

<3>涉及数据库的操作，经常出错，我一般都加个try-catch

（3）sql返回值

单次insert成功时返回1，失败0

select成功时返回值可自定义（resultType=java.lang.*)，失败会返回null

delete的返回值可在mapper里定义，一般为0，1或true,false或void

update返回值为记录数。
